# Web app (PWA) — install on iPhone, iPad, Android & desktop for free

> **Goal.** Run FieldMark on **iOS / iPadOS** (and every other device) **without an
> Apple Developer account, without the App Store, and without any fee** — straight
> from the browser. This is done by shipping the existing app as a **PWA**
> (Progressive Web App): a normally-hosted web page that Safari/Chrome can
> **install to the home screen**, where it runs full-screen and **fully offline**.

This is the free path to iOS/iPadOS. It reuses the *exact* renderer that already
ships on Windows, macOS and Android — the cross-platform rule in `CLAUDE.md`
(“Android runs the desktop renderer verbatim”) means the web app is the same
bundle plus PWA packaging. No feature is rebuilt.

---

## What you get

- **Installable** on iOS/iPadOS Safari, Android Chrome, and desktop Chrome/Edge —
  an app icon on the home screen, launches full-screen (no browser chrome).
- **Fully offline.** A service worker precaches the whole bundle on first load, so
  after that the app opens and works with **no network at all** — keeping the
  app’s “offline, no cloud, no telemetry” promise.
- **The majority of features** — view, organize/merge pages, mark up, measure by
  scale, fill forms, number/stamp, cosmetic signatures — all run in the browser
  renderer. See *Feature parity & limits* below for the few file-I/O nuances.

## How it’s built

```bash
npm run build:pwa    # assembles dist-pwa/ (app + manifest + service worker + icons)
npm run verify:pwa   # build + assert it's a valid, installable PWA (CI runs this)
```

`scripts/build-pwa.js` takes the shared `www/` bundle produced by
`scripts/build-web.js` (the same one Capacitor loads on Android), copies it to
`dist-pwa/`, and layers on:

- `manifest.webmanifest` — name, icons, `display: standalone`, theme color.
- `sw.js` — an offline service worker that precaches every bundle asset
  (cache-first for same-origin; cross-origin requests such as the optional GitHub
  update check are left to the network and fail gracefully offline).
- iOS `apple-mobile-web-app-*` meta tags + an `apple-touch-icon`.
- `.nojekyll` so GitHub Pages serves the files as-is.

Icons are generated by `build/make-pwa-icons.js` from the **same artwork** as the
desktop and Android launchers, so the identity is consistent everywhere.

`dist-pwa/` is **host-agnostic static files** — it needs only HTTPS (required for
service workers and “Add to Home Screen”). It works from a domain root or a
project sub-path (all paths are relative), so GitHub Pages’ `/PDF-Reader/` URL is
fine.

## Hosting on GitHub Pages (free HTTPS)

A workflow is included at `.github/workflows/pages.yml`.

**One-time setup (repo owner):**
1. GitHub → **Settings → Pages → Build and deployment → Source: “GitHub Actions.”**
2. Merge to `main` (or run the **Deploy PWA to GitHub Pages** workflow manually via
   *Actions → Run workflow*).

The workflow builds `dist-pwa/`, verifies it, and publishes it to:

```
https://killswit3h.github.io/PDF-Reader/
```

Every later push to `main` republishes automatically.

> Prefer a different host? `dist-pwa/` drops straight onto Netlify, Vercel,
> Cloudflare Pages, or any static HTTPS server — no config needed.

## Installing on a device

### iPhone / iPad (Safari — the free iOS path)
1. Open the hosted URL in **Safari** (must be Safari; other iOS browsers can’t add
   PWAs to the home screen).
2. Tap the **Share** button → **Add to Home Screen** → **Add**.
3. Launch it from the new icon. It opens full-screen and works offline.

### Android (Chrome)
- Open the URL → Chrome shows an **Install app** prompt (or menu → **Install app /
  Add to Home screen**).

### Desktop (Chrome / Edge)
- Open the URL → click the **Install** icon in the address bar.

## Feature parity & limits (honest)

The renderer is identical across platforms; only **file I/O** differs, and it’s
already abstracted behind `window.api` (`src/renderer/js/platform-web.js`):

- **Open** a PDF → the system file picker (Files app on iOS, file dialog on
  desktop). Fully supported.
- **Save** → the browser downloads the finished PDF (to *Files* / *Downloads*),
  since a web app can’t silently overwrite a picked file in place. Same behavior
  the app already uses in a plain browser today — Save routes to “Save As”.
- **In-app self-update** is a desktop/Android concept; the PWA updates itself
  automatically when you reload after a new deploy (the service worker swaps in the
  new version).

iOS PWA caveats worth knowing: storage can be evicted by the OS under heavy
pressure, and there’s no true persistent file handle — but the document data lives
in the PDF you open/save, not in browser storage, so this doesn’t risk your work.

## “Can it be a real Safari App Store app / Safari extension later?”

Yes, but that path is **not free** and is **not needed** for the goal above:

- A **Safari web extension** or a **native App Store app** must be wrapped in a
  macOS/iOS project with **Xcode**, **signed**, and distributed through the App
  Store or a signed build — which requires the **paid Apple Developer Program
  ($99/yr)** and a Mac. (The repo already has an iOS/TestFlight pipeline under
  `fastlane/` and the `ios:*` npm scripts if you ever go that route.)
- Because the app is already a PWA, converting it to a Safari web extension later
  is mostly mechanical: `xcrun safari-web-extension-converter` wraps a web
  bundle + an extension `manifest.json` into an Xcode project you then sign. The
  same `dist-pwa/` payload feeds it. **This step can only be run on macOS** — it
  can’t be produced in a Linux CI environment.

**Recommendation:** ship the PWA (free, installs on iOS/iPad/Android/desktop
today); only pursue the paid App Store/extension route if you specifically need
App Store distribution or a capability PWAs can’t offer.
